対象: $ARGUMENTS

## Step 1: コンテキストの収集

- `docs/PRODUCT_CONTEXT.md` を読む
- `.claude/epics/` 配下に既存のエピックファイルがあれば読む

## Step 2: 壁打ち

以下の観点でユーザーに質問・指摘し、全体像を詰める。質問は一度にまとめて行い、回答を待つ。

- プロダクトのユーザーは誰か（ロール・権限）
- 必須機能と nice-to-have の区別
- 外部サービス・既存システムとの連携
- リリース優先度・フェーズ分けの考え方

## Step 3: エピック分割

- 各エピックはユーザーに価値を提供できる独立した機能領域とする
- エピック間の依存関係を整理し、並列可否を判断する
- 分割案をユーザーに提示し、確認を求める

## Step 4: ファイルの作成

承認後に以下のファイルを作成する（`.claude/epics/` はgit管理）。

```
.claude/epics/
├── index.md       # エピック一覧・依存関係・並列可否テーブル
├── epic_A.md
└── epic_B.md
```

### index.md

```markdown
# [プロダクト・機能名] エピック一覧

## 概要

[全体の概要]

## エピック一覧

| ID  | エピック名 | 概要   | 依存 | 並列可否 |
| --- | ---------- | ------ | ---- | -------- |
| A   | [名前]     | [概要] | なし | ✅       |
| B   | [名前]     | [概要] | A    | ❌       |

## 実行順序

[依存関係をもとにした推奨実行順序]
```

### epic\_[ID].md

```markdown
# エピック[ID]: [エピック名]

## 概要

[エピックの目的・ユーザーへの価値]

## 含まれるストーリー（想定）

- [ストーリーの概要1]
- [ストーリーの概要2]

## 依存エピック

[依存するエピックのID・理由、なければ「なし」]
```

## 補足

### 用語定義

| 用語           | 定義                                                                                                                               | 粒度 |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ---- |
| **エピック**   | 複数のストーリーをまとめた大きな機能領域のまとまり。ユーザーに独立した価値を提供できる単位。必ず複数のストーリーに分解して実装する | 大   |
| **ストーリー** | 「[ロール]として、[目的]のために、[機能]したい。」の形式。数日で完了できる単位。1PR = 1ストーリー                                  | 小   |

#### 具体例（ECサイトの場合）

```
エピック: ユーザー認証
├── ストーリー1: ユーザーとして、購入履歴を確認するために、メールアドレスとパスワードでログインしたい
├── ストーリー2: ユーザーとして、商品を購入できるようにするために、新規アカウントを登録したい
└── ストーリー3: ユーザーとして、ログインできなくなったときに備えるために、パスワードをリセットしたい
```

### 並列可否の判断

**並列化はエピック単位が最大粒度。エピック内での並列化は行わない。**

| 単位                       | 並列可否  | 理由                                                     |
| -------------------------- | --------- | -------------------------------------------------------- |
| エピック間（独立）         | ✅ 推奨   | ブランチ・worktreeが分離されており競合しない             |
| エピック内（ストーリー間） | ❌ 非推奨 | 同一ファイルへの同時編集によるコンフリクトが発生しやすい |

この時点の並列可否は**論理依存（機能として先後関係があるか）をもとにした暫定判断**である。実装依存（同じファイルやスキーマを変更するか）は各エピックの `/spec` に入るまでわからないため、並列開始前に確認すること。
